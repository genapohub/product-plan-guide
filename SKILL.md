# 产品经理产品方案产出指南

## Overview

本技能将一份结构化的产品经理方法论转化为可执行的工作流。当用户提出任何产品需求或方案产出任务时，先识别该需求属于 5 类场景中的哪一类，再按对应场景的产出清单和通用模板生成完整、可执行的产品文档。

详细的方法论、各场景产出清单、文档核心要点、通用模板、提示词模板、质量检查清单均存放在 `references/产品方案产出指南.md`，在执行前必须读取对应章节。

## 触发条件

当出现以下任一情况时触发本技能：

- 用户提出一个产品需求或想法，需要产出方案文档
- 用户提到"写PRD""产品方案""需求规划""功能设计""需求文档"等关键词
- 用户描述一个业务场景，需要判断该做什么程度的产品工作
- 用户要求做市场调研、竞品分析、可行性分析、埋点方案、上线方案
- 用户说"这个需求怎么做""帮我规划一下""出个方案"

## 执行流程

按以下 7 步顺序执行，不可跳步。

### Step 0: 加载历史记忆

在执行本技能前，先检查并加载历史记忆：

1. 检查项目目录下是否存在 `.skill-memory/product-plan-guide/` 目录，如不存在则创建
2. 读取 `.skill-memory/product-plan-guide/MEMORY.md`（如果存在）——了解长期产品决策、商业模式约定、竞品格局、技术栈偏好等
3. 读取 `.skill-memory/product-plan-guide/YYYY-MM-DD.md`（今日日志，如果存在）——了解近期已完成的产品方案

加载记忆后，在回复中简要提及相关的历史上下文（如"根据之前定义的商业模式，该产品采用三级定价..."），让用户感受到连贯性。

### Step 1: 需求理解

### Step 1: 需求理解

- 解析用户输入的需求描述
- 提取关键信息：业务背景、目标用户、功能范围、预期周期、是否全新产品
- 识别缺失的关键信息，主动向用户提问补全（一次最多问 2-3 个最关键的问题，避免轰炸式提问）

### Step 2: 场景识别

读取 `references/产品方案产出指南.md` 的"一、场景识别"章节，用下表判断需求属于哪类场景：

| 场景 | 名称 | 判断条件（满足任一） | 产出量 |
|------|------|---------------------|--------|
| 场景一 | 0到1新项目 | 全新产品线/新业务方向、无现有产品基础、需从市场调研开始、涉及商业模式设计 | 12类 |
| 场景二 | 中大型需求迭代 | 已有产品新增功能模块、涉及多页面/多角色、开发周期2周以上、需跨团队协作 | 6-8类 |
| 场景三 | 小优化/紧急修复 | 单一功能点优化、Bug修复或体验改进、周期1周以内、不涉及新业务逻辑 | 2-3类 |
| 场景四 | 大版本功能升级 | 现有产品重大版本迭代(如V2.0)、涉及核心流程改造、可能改变产品定位、涉及用户迁移 | 8-10类 |
| 场景五 | 探索性/预研项目 | 业务方向未定、需验证可行性、可能不立项、输出为决策依据 | 3-4类 |

识别流程：是否全新产品？→ 场景一；是否大版本升级？→ 场景四；是否方向探索？→ 场景五；是否多页面多角色？→ 场景二；否则 → 场景三。

### Step 3: 与用户确认场景

输出场景判断结果，包含：
1. 场景判断：场景X（场景名称）
2. 判断依据：逐条对照说明
3. 建议产出清单：列出将生成的必备文档
4. 预估周期：X 天/周/月

向用户确认："判断为场景X，将产出以下文档：...，是否确认？如有补充信息请告知。" 等用户确认后再进入产出阶段。用户可能补充信息导致场景变更，需重新判断。

### Step 4: 按清单产出文档

读取 `references/产品方案产出指南.md` 中对应场景的章节，获取：
- 该场景的必备产出清单（文档名称、阶段、优先级）
- 每个文档的核心要点
- `references/产品方案产出指南.md` 的"七、通用文档模板"章节提供的模板格式

产出要求：
- 按阶段顺序生成：调研 → 规划 → 设计 → 落地 → 验证
- 使用通用模板的格式（PRD/BRD/MRD/埋点方案/上线方案模板）
- 每个文档内容完整、逻辑清晰、可执行
- 每段不少于 3-5 句，每节不少于 150-200 字，避免空洞
- 文档间逻辑一致，无矛盾
- 产出后保存为 Markdown 文件，命名规范：`{项目名}_{场景编号}_{文档类型}_{版本号}.md`

### Step 5: 质量检查

读取 `references/产品方案产出指南.md` 的"九、产出质量检查清单"章节，对照检查：

- 场景判断是否正确并已与用户确认
- 产出清单是否完整（必备文档全部生成）
- 文档格式是否规范（Markdown、层级清晰）
- 内容深度是否达标（每段3-5句、每节150-200字）
- 逻辑是否连贯（文档间无矛盾）
- 是否可执行（能直接用于开发/测试/上线）
- PRD 专项：功能清单完整、每个功能含输入/处理逻辑/输出/交互/异常、流程图含主流程/异常/分支、埋点用表格
- 上线方案专项：灰度策略明确、回滚方案可执行、监控告警已定义、应急预案含分级
- 数据方案专项：北极星指标单一、KPI已拆解、埋点覆盖关键路径、A/B测试方案完整

识别遗漏内容并补充，最后输出交付清单（列出所有生成的文档及路径）。

### Step 6: 记录执行记忆

完成实质性产品方案产出后，将关键信息持久化到项目记忆目录：

1. **写入今日日志** — 向 `.skill-memory/product-plan-guide/YYYY-MM-DD.md` **追加**（不存在则先创建）：
   - 场景类型（场景X）
   - 产品/项目名称
   - 产出文档清单
   - 关键产品决策（如商业模式、定价策略、核心功能优先级、技术栈方向等）
   - 待后续跟进的事项

2. **更新长期记忆** — 如果产生了**可复用的产品知识或决策**（如商业模式假设、竞品格局更新、用户画像发现、定价策略、Roadmap规划等），同步更新 `.skill-memory/product-plan-guide/MEMORY.md`：
   - 按主题分类（商业模式 / 竞品格局 / 用户画像 / 产品策略 / Roadmap）
   - 保持简洁，每条 1-2 行

**只需记录实质性工作**（产出了PRD/BRD/MRD/竞品分析/用户研究等产品文档）。纯问答、简单咨询不需记录。

3. **同步到仓库** — 可将 `.skill-memory/product-plan-guide/` 下的变更提交到 Git，实现团队共享记忆。

**记忆目录结构：**
```
项目目录/.skill-memory/product-plan-guide/
├── 2026-07-22.md
└── MEMORY.md
```

### references/产品方案产出指南.md

完整的方法论文档，包含：
- 场景识别决策表与流程图
- 5个场景的详细产出清单与各文档核心要点
- 通用文档模板（PRD/BRD/MRD/埋点方案/上线方案）
- AI 工具使用建议（提示词模板、工作流）
- 产出质量检查清单

在 Step 2（场景识别）、Step 4（产出文档）、Step 5（质量检查）时必须读取对应章节。文档较大（约 700 行），按需读取对应章节而非全量加载：
- 场景识别：读取"一、场景识别"
- 产出文档：读取对应场景章节（二至六）+ "七、通用文档模板"
- 质量检查：读取"九、产出质量检查清单"

## 注意事项

- 不要跳过 Step 3 的用户确认环节，直接产出可能方向错误导致返工
- 用户信息不足时主动提问，但控制提问数量（单次不超过 3 个）
- 场景三（小优化）不要过度产出，2-3 类文档足够，避免形式主义
- 场景一（0到1）和场景四（大版本）产出量大，建议分轮次生成，每轮确认后再继续
- 模板是骨架不是枷锁，结合具体业务灵活填充，避免填空式空洞内容
